# Ayama Separation — Implementation Strategies

Status: `designed` · Type: **Planning / design (Explanation)**. Tier **2** — the sibling
[`AYAMA_SEPARATION_RISK_REGISTER.md`](AYAMA_SEPARATION_RISK_REGISTER.md) gates every phase
below; the [`AYAMA_SEPARATION_MASTER_PLAN.md`](AYAMA_SEPARATION_MASTER_PLAN.md) holds the why,
the measured starting state, and the phase table. Risk IDs are cited at each edit site that
exists to mitigate them.

---

## S1 — Copy, never move (Phase 1) · mitigates `DR1`, `DR2`

```
robocopy G:\phyriad\apps\ayama G:\phyriad\projects\Ayama /E /XD build
```

`apps/ayama` is left **byte-for-byte intact**. `build/` is excluded — it is a local artifact
(and it embeds absolute `G:/Phyriad/apps/ayama/...` paths that would be wrong at the new
location anyway).

**Verification (`DR2`):** compare recursive file counts, source vs destination, excluding
`build/`. A mismatch halts the phase. The source count at design time is 111 files / 1.2 MB.

## S2 — Vendor the closure (Phase 2) · mitigates `SR1`, `DR5`

**Edit site: `projects/Ayama/CMakeLists.txt:53`.**

```cmake
# before — resolves to the container root
set(PHYRIAD_ROOT "${CMAKE_CURRENT_SOURCE_DIR}/../..")
# after — resolves to Ayama's own vendored copy
set(PHYRIAD_ROOT "${CMAKE_CURRENT_SOURCE_DIR}")
```

with the vendored tree laid out so that `${PHYRIAD_ROOT}/catalog/cpp/<sub>` keeps resolving —
i.e. the `ensure_phyriad_target` macro at `:55-60` and its 18 call sites at `:62-96` need **no
edit**. Two shapes are possible and the choice is deferred to build time:

- **Shape A (zero macro edits):** vendor to `projects/Ayama/catalog/cpp/<pillar>/`.
  `PHYRIAD_ROOT = ${CMAKE_CURRENT_SOURCE_DIR}`. Nothing else in the file changes.
- **Shape B (PhyriadFG-shaped):** vendor to `projects/Ayama/framework/<pillar>/` and rewrite
  the macro's `add_subdirectory(${PHYRIAD_ROOT}/catalog/cpp/${_sub} …)` to
  `${PHYRIAD_ROOT}/framework/${_sub}`. One macro line changes; the tree reads like PhyriadFG's.

**Shape B is preferred** — the master plan §5 target names `framework/`, and PhyriadFG uses that
name. Shape A is the fallback if B surfaces a path assumption inside a vendored pillar's own
CMakeLists.

**The vendoring set** (master plan §4; grep-derived, linker-settled):

```
hal schema transport topology scheduler node graph tuning process etw stigmergy ipc _meta
ui render render/present render/opengl3 render/vulkan   (ayama-ui only)
```

**`render/vulkan/bench` is excluded** (`DR5`) — 275 MB of FG bench artifacts, no Ayama consumer.
Vendor `render/vulkan/{src,include,shaders,CMakeLists.txt}` only; if `render/vulkan/CMakeLists.txt`
references `bench/` unconditionally, guard the reference with `if(EXISTS …)` rather than
vendoring the directory.

**The unresolved set** — `runtime`, `orchestration`, `behavior`, `correlation`, `profile` are
**not** vendored initially (zero `#include` sites). If the link fails, they are added and the
master plan §4 finding is corrected in the same change (`DR3`).

> **Outcome (2026-07-16):** the set resolved during implementation, before the first build —
> reading `ui/CMakeLists.txt` first-hand showed `phyriad_ui` hard-links `phyriad_runtime`
> (which drags `graph`, `profile`, `scheduler`) — so **`runtime` and `profile` joined the
> vendored set (17 units total)**. `orchestration`, `behavior`, `correlation` stayed out and
> the link is green without them (isolated from-scratch build, 49/49). Shape B landed as
> preferred. Per-unit rationale: `framework/VENDORED.md`.

**Verification:** a from-scratch configure + build in a fresh directory. Green is the gate. Note
`ensure_phyriad_target`'s known failure mode, documented in the file's own comment at `:78-79`:
an **undefined target silently degrades to a raw `-l` flag, so configure passes while includes
are absent**. Configure-passes is therefore NOT the gate — a full build + link is.

## S3 — Repair the false green (Phase 3) · mitigates `TR1`

**Edit site: `projects/Ayama/CMakeLists.txt`, after the `project()` call at `:17`.**

```cmake
include(CTest)          # generates the root CTestTestfile.cmake → ctest discovers subdirs
```

Without a root `CTestTestfile.cmake`, ctest reports `No tests were found!!!` **and exits 0**, so
`testbench.ps1`'s `$LASTEXITCODE` check prints `[ayama] OK` having run nothing. This is the
defect that hid `TR2` for ~9 weeks.

**Edit site: `projects/Ayama/testbench.ps1`.** The script is generated by
`scripts/gen_testbench.py` and passes `-DPHYRIAD_BUILD_TESTS=ON`, but Ayama's option is
`AYAMA_BUILD_TESTS` (`CMakeLists.txt:24`, default `ON` — which is why tests built anyway). Pass
both, and **fail on an empty test set** rather than trusting ctest's exit code:

```powershell
cmake -S $PSScriptRoot -B $build -DPHYRIAD_BUILD_TESTS=ON -DAYAMA_BUILD_TESTS=ON -DCMAKE_EXPORT_COMPILE_COMMANDS=ON
...
# ctest exits 0 on an empty set — assert the set is non-empty, else this is a false green
$found = (ctest --test-dir $build -N | Select-String "Total Tests: (\d+)").Matches.Groups[1].Value
if ([int]$found -lt 1) { Write-Host "[$unit] NO_TESTS_DISCOVERED"; exit 4 }
```

The generated-file header says *"Regenerate, don't hand-edit"* — that rule belongs to the
container's `scripts/gen_testbench.py`, which does not follow Ayama to `projects/`. Once
separated, Ayama's `testbench.ps1` is Ayama's own file; the header is updated to say so.

## S4 — Repair the time-bomb test (Phase 3) · mitigates `TR2`

**Edit site: `projects/Ayama/learn/tests/learn_test.cpp:246-251`.**

The test hardcodes `"2026-05-14T12:00:00Z"` as the "fresh" entry while
`PerGameMemory::needs_revalidation` (`learn/src/PerGameMemory.cpp:646-671`) measures age against
**today** via `GetLocalTime`. Any hardcoded date fails once wall-clock passes it — the fix must
remove the dependency on when the test runs, not merely move the date forward:

```cpp
// Derive the "fresh" timestamp from today, so the test cannot age out.
// A hardcoded date passes when written and fails N days later — that is what
// happened to "2026-05-14T12:00:00Z" (this test aborted at line 251 from ~2026-06-13 on,
// invisible behind the ctest false green — see S3).
SYSTEMTIME st{}; GetLocalTime(&st);
char today_iso[21];
std::snprintf(today_iso, sizeof(today_iso), "%04d-%02d-%02dT12:00:00Z",
              st.wYear, st.wMonth, st.wDay);
std::strncpy(good.last_validated, today_iso, sizeof(good.last_validated) - 1);
```

The stale entry keeps its hardcoded `"2020-01-01T00:00:00Z"` — a date in the fixed past cannot
age *out* of a >30-day window, only deeper into it, so it is safe by construction.

**Verification:** `ayama_learn_test.exe` exits **0** (it currently exits 127). Then the full
ctest set: **10 discovered, 10 passing**. A green that has run nothing does not count.

**Not fixed here:** `config/` has an `add_test` at `config/CMakeLists.txt:47` behind
`if(EXISTS tests/config_test.cpp)`, and that source does not exist — so `ayama_config_test` has
never existed. Writing it is out of this migration's scope; it is recorded as an honest coverage
gap in the master plan §8.

## S5 — The front door (Phase 4)

- **`README.md`** — repair the 6 dead links (`../../{README,LICENSE,FAQ,ARCHITECTURE}.md`,
  `../../docs/WINDOWS_DISTRIBUTION.md`, `../../docs/ayama/reports/…`). They pointed at a
  container root that no longer has those files; the targets now live inside `projects/Ayama/docs/`.
  The "showcase application for the Phyriad Framework" framing is rewritten: Ayama is an
  independent project that vendors a substrate copy, not a showcase for the container.
- **`LICENSE`** — Ayama's existing PolyForm Noncommercial 1.0.0, copied in (PhyriadFG carries its
  own LICENSE at its root; note PhyriadFG's is MIT — **do not** copy PhyriadFG's license text).
- **`build.bat` / `build-release.bat`** — PhyriadFG-shaped entry points wrapping the CMake
  invocation. Ayama's existing `tools/ayama-ui/build.bat:27` sets `FRAMEWORK=%ROOT%\framework`
  against a path that has not existed since the reorganization — after S2 that variable becomes
  meaningful again, pointing at the vendored tree.
- **`docs/`** — import the 17 files from `catalog/cpp/docs/apps/ayama/` (README, FAQ,
  EMPIRICAL_TEST_PROTOCOL, `reports/` with the 12 per-game reports + `raw_data/`'s 11 CSVs) and
  the 10 planning docs from `catalog/cpp/docs/plans/archive/ayama/`. **Copy, not move** — the
  originals are removed in Phase 7 with the rest (`DR1`).
- **`ReportExporter.hpp:51-53,74`** — the hardcoded runtime path `..\..\..\..\docs\ayama\reports`
  is re-pointed at the new `docs/reports/`. It is broken today regardless of this migration.

## S6 — The decoupling proof (Phase 5) · mitigates `SR1`, `DR3`

The only honest evidence that vendoring worked. Configure and build Ayama from a fresh directory
**with `catalog/cpp` renamed out of the way**:

```
mv G:\phyriad\catalog\cpp G:\phyriad\catalog\cpp__hidden
cd G:\phyriad\projects\Ayama && .\testbench.ps1
mv G:\phyriad\catalog\cpp__hidden G:\phyriad\catalog\cpp     # ALWAYS restore
```

Green with the substrate hidden = self-contained. Any failure names a pillar the closure missed
→ vendor it → repeat. A build that merely succeeds with `catalog/cpp` present proves nothing:
`../..` would still resolve.

**`DR4` — the rename touches the substrate other projects build against.** The restore runs
unconditionally, including on failure, and no other build may run during the window. If the
session dies mid-window, `catalog/cpp__hidden` is the recovery marker: rename it back.

## S7 — git init (Phase 6)

`git init` + an initial commit of the whole tree. Standing rule 2: the container has no git;
child projects carry their own. No remote is configured — publishing is the operator's call, and
PolyForm Noncommercial is not the same posture as PhyriadFG's public MIT repo.
A `.gitignore` covering `build/`, `build-release/`, `_deps/` is written (PhyriadFG's is the model).

> **Outcome (2026-07-16, operator's order):** both defaults were superseded — the operator
> chose **MIT** (PhyriadFG's exact license) and a **public GitHub repository**. The LICENSE,
> README and FAQ were re-licensed before the initial commit; nothing was ever published under
> PolyForm.

## S8 — Container reconciliation (Phase 7, operator-gated) · mitigates `DR1`, `DR3`

**Only after Phase 5 is green and the operator gives an explicit go.** The 7 touchpoints from
the master plan §7 are updated **in the same change** as the removal of `apps/ayama` — a
touchpoint updated early points at a project that does not exist yet; one updated late points at
a directory that no longer does. Either way, one of them is a lie (BOOT §3 rule 4).

Order within the phase: update the 7 touchpoints → verify each → **then** remove `apps/ayama`.
`.clangd` is **regenerated** via `scripts/gen_clangd.py`, not hand-edited. `catalog/cpp/INVENTORY.md`
loses its `## apps/ayama/` section and `scripts/check_inventory.py` must pass afterward.
